01 · Overview
Executive Summary
The Playbook serves three primary audiences and defines standard governance for deploying digital models across the Large Molecule modality.
Purpose

To provide sites, leaders and project teams with the enterprise endorsed framework that establishes mandated, platform-aligned digital capabilities across all sites and provides an evidence-based, value-focused framework for digital process intelligence adoption.

For Business Leaders

Govern platform-aligned digital investments to maximise ROI and scale evidence-based value across the network.

For Sites

Deploy mandated digital capabilities using standardised guidance to deliver proven operational value.

For Project Teams / CMC / Molecule Stewards

Embed mandated digital capabilities into CMC development plans with defined work packages, resources, timelines and transfer readiness.

Value & Impact
Maximise digital ROI across the enterprise
Reduce duplication of effort and local reinvention
Minimise risk and cost through standardisation
Enable faster, more confident decision-making

10 · Model Lifecycle
Process Pulse Model Lifecycle
Three-stage lifecycle covering data foundations, development & assurance, and deployment & execution.
Assumptions: RAMAN installation complete at site · Process Pulse configuration in place

① Data Foundations

1a. Data Collection
Where is data generated?
Is it GxP compliant?
Is collection structured for modelling?
1b. Data Transfer
What network layer is data on?
Is secure transfer in place?
Does it integrate with existing pipeline?

② Model Development & Assurance

2a. Model Development
Does the model reflect modality-level process assumptions?
Are 2k/5k scale differences accounted for?
Is design aligned to a defined value case?
2b. Model Development Report
Are assumptions, inputs, outputs documented?
Are performance metrics defined?
2c. Model Validation Report
Has site-level validation been completed?
Is change control clearly defined?

③ Deployment & Execution

3a. Model Implementation
Where will the model sit (DCS, Process Pulse, dashboard)?
Does integration fit existing workflows?
3b. Model Execution Protocol
When should the model be used?
What decisions does it inform?
What is the escalation pathway if outputs conflict?
3c. Model Execution Report
Are yield and throughput improvements tracked?
Is value sustained and visible?

What is MoSCoW?

A structured prioritisation framework used to categorise requirements based on criticality and value. In DPI it helps sites and leaders determine which capabilities are mandatory, recommended, or optional.

Why use it?
Focus on proven, value-delivering capabilities
Alignment with mandated, platform-based standards
Efficient allocation of time, cost and resources
Transparent trade-offs during planning
